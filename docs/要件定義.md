--🚀スマート勤怠管理システム：要件定義書🚀--

##1. システム概要
本システムは、AI（顔認証技術）を活用した非接触型の「スマート勤怠管理システム」である。出入り口に設置した端末（Webブラウザ/将来的なデスクトップアプリ）からリアルタイムで従業員を識別し、出退勤ログを自動記録すると同時に、LINEを介して本人へリアルタイム通知を行う。また、人事担当者向けにデータ可視化および管理が可能なWebダッシュボードを提供する。

----

##2. 機能要件
システムが提供すべき具体的な機能を、アクター（利用者・システム）別に定義する。

### 2.1 従業員向け機能（エッジ端末・打刻画面）

| 機能ID | 機能名 | 機能概要 | 優先度 |
| :--- | :--- | :--- | :--- |
| **F-1-1** | カメラ映像取得 | 端末のカメラからリアルタイムのビデオストリームを取得する。 | 必須 (Must) |
| **F-1-2** | 自動顔検知・送信 | カメラ前に立った人物の顔を自動で検知（キャプチャ）し、バックエンドAPIへ送信する。 | 必須 (Must) |
| **F-1-3** | 打刻結果フィードバック | 認識結果（成功：緑、遅刻：黄、未登録：赤）を画面上に視覚的・直感的に表示する。 | 必須 (Must) |

### 2.2 人事・管理者向け機能（管理者Webダッシュボード）
| 機能ID | 機能名 | 機能概要 | 優先度 |
| :--- | :--- | :--- | :--- |
| **F-2-1** | 新規従業員登録 | 社員ID、氏名、部署、**メールアドレス**、および**顔写真（3〜5枚）**を一括登録する。 | 必須 (Must) |
| **F-2-2** | 従業員情報管理 | 登録済み従業員の情報の編集、および退職者のデータ（顔データ含む）の削除。 | 必須 (Must) |
| **F-2-3** | リアルタイム状況監視 | 当日の打刻ログ（誰が何時に来たか）を最新順にリアルタイムで一覧表示する。 | 必須 (Must) |
| **F-2-4** | 勤怠統計の可視化 | **Chart.js** を用い、出勤率や遅刻率の統計をグラフで可視化する。 | 必須 (Must) |
| **F-2-5** | データエクスポート | 指定期間の勤怠データを**CSVまたはExcel形式**でダウンロードできる。 | 必須 (Must) |

### 2.3 バックエンド・AI処理機能
| 機能ID | 機能名 | 機能概要 | 優先度 |
| :--- | :--- | :--- | :--- |
| **F-3-1** | 顔特徴量抽出 (AI) | **InsightFace** を用い、画像から顔の512次元ベクトル（Embedding）を抽出する。 | 必須 (Must) |
| **F-3-2** | 顔識別・照合 | 抽出したベクトルと **PostgreSQL** 内のデータを照合（または事前に学習させたSVMで分類）し、個人を特定する。 | 必須 (Must) |
| **F-3-3** | ビジネスロジック判定 | サーバー時刻と既定の始業時刻を比較し、「遅刻」「早退」「正常」を自動判定する。 | 必須 (Must) |
| **F-3-4** | メール自動通知連携 | 打刻完了時、該当従業員のメールへ結果をプッシュ通知する。 | 必須 (Must) |

----

## 3. 非機能要件 (Non-Functional Requirements)
システムの品質、性能、セキュリティに関する基準を定義します。

### 3.1 性能・応答性 (Performance)
*   **処理速度：** 従業員がカメラの前に立ってから、顔識別、遅刻判定、DB保存、画面への結果表示（フィードバック）までの一連の処理を**2秒以内**に完了させること。
*   **識別精度：** 適切な照明条件下において、誤認識率（他アカウントへの誤打刻）を実用レベル以下に抑えるとし、しきい値を調整可能とすること。

### 3.2 セキュリティ (Security)
*   **認証・認可：** 管理者ダッシュボードへのアクセスは、ID/パスワードによる認証（JWTトークン等）を必須とし、一般従業員によるアクセスを遮断すること。
*   **個人情報保護：** 顔写真および顔の特徴量ベクトル（個人識別情報）は、暗号化を考慮し、外部からの直接アクセスを禁止すること。

### 3.3 可用性・運用環境 (Availability & Infrastructure)
*   **コンテナ化：** 環境依存のトラブルを防ぐため、バックエンド環境はすべて **Docker** でコンテナ化すること。
*   **クラウド配置：** フロントエンドは **Vercel**（またはGitHub Pages）、バックエンドおよびデータベースは **Render** にデプロイし、24時間アクセス可能とすること。
*   **CI/CD：** **GitHub Actions** を活用し、`main` ブランチへのコードプッシュ時に自動ビルド・デプロイが行われること。
*   **拡張性：** 将来的な「デスクトップアプリ化」を見据え、フロントエンド（React）とバックエンド（FastAPI）はAPIを介して通信する**疎結合アーキテクチャ**を採用すること。
