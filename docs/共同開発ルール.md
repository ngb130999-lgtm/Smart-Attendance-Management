# 共同開発ルール

チームで開発を円滑に進めるためのルールをまとめたドキュメントです。プロジェクト開始時にメンバー全員で内容を確認し、必要に応じて合意の上で修正してください。

---

## 1. 基本方針

- 「動くコード」より「読めるコード」を優先する。
- ルールは絶対ではなく、チームの合意で随時見直してよい。

---

## 2. Git / ブランチ運用

### 2.1 ブランチ戦略

| ブランチ | 用途 | ルール |
|---|---|---|
| `main` | 常に動く状態を保つ本番用 | 直接pushは禁止。PR経由のみ |
| `Issue番号-feature/機能名` | 機能開発用 | 1機能につき1ブランチ |
| `Issue番号-fix/内容` | バグ修正用 | 修正内容が分かる名前にする |

- ブランチ名は英小文字とハイフンで統一する（例: `feature/login-form`）。
- 作業開始前に必ず `git pull` して最新を取り込む。
- 作業中のブランチが長期化しないよう、こまめにマージする（目安: 3日以内）。

### 2.2 コミットルール

- コミットは「1つの変更につき1コミット」を意識する。
- コミットメッセージの書式（例）:
  - `feat: ログイン画面のUIを追加`
  - `fix: 投稿削除時のエラーを修正`
  - `docs: READMEにセットアップ手順を追記`
  - `refactor: 重複コードを関数化`
- `WIP` や `修正` だけのメッセージは避ける。

### 2.3 コンフリクト対応

- コンフリクトが発生したら、関係するコードを書いた本人に確認してから解消する。
- 勝手に相手の変更を消さない（`--theirs` / `--ours` の安易な使用に注意）。
- 解消に自信がない場合はチームに共有してから進める。

---

## 3. プルリクエスト（PR）とレビュー

- `main` へのマージは必ずPRを作成し、**最低1人のレビュー承認**を得る。
- PRには以下を書く:
  - 何を変更したか
  - なぜ変更したか
  - 動作確認の方法・結果
- レビューコメントは「コードへの指摘」であり「人への指摘」ではないことを全員が意識する。

---

## 4. コーディング規約

- 使用言語の標準的なスタイルガイドに従う（例: PythonならPEP 8、JavaScriptならPrettier + ESLint）。
- フォーマッタ・リンタを導入し、コミット前に実行する。
- 変数名・関数名は英語で、意味の分かる名前をつける（`a`, `tmp2` などは避ける）。
- マジックナンバーは定数化する。
- コメントは「何をしているか」より「なぜそうしたか」を書く。

---

## 5. Issue / タスク管理

- タスクはすべてIssue（またはタスク管理ツール）に登録し、口頭だけで済ませない。
- Issueには担当者・期限を必ず設定する。
- 着手したらステータスを「進行中」に変更し、誰が何をしているか見える状態を保つ。
- 期限に間に合わなそうな場合は、**期限当日ではなく分かった時点で**報告する。

---

## 6. コミュニケーション

- 定例ミーティングを設定する（例: 週1回、進捗・課題・次の予定を共有）。
- 決まったことは必ず議事録・ドキュメントに残す。「言った・言わない」を防ぐ。

---

## 7. 環境・セキュリティ

- 開発環境のセットアップ手順を `README.md` に記載し、誰でも再現できるようにする。
- 依存関係はファイルで管理する（`requirements.txt` / `package.json` など）。

---

## 8. ドキュメント

- 以下は最低限メンテナンスする:
  - `README.md`: プロジェクト概要、セットアップ手順、起動方法
  - 設計資料: DB設計、API仕様、画面構成
- 仕様変更があったらドキュメントも同じPRで更新する。

---

## 9. トラブル・意見が割れたときの決め方

1. まず各自の意見と理由を出し合う。
2. 技術的な判断は、実際に小さく試して比較できるならそれで決める。
3. 決まらない場合は多数決、または担当領域のリーダーが決定する。
4. 一度決めたことには全員が従う（後から蒸し返さない）。ただし問題が判明したら再協議してよい。
